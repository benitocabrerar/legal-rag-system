2. Análisis Detallado del Problema

2.1 Contexto Inicial

El sistema requería agregar columnas adicionales a las tablas de finanzas (finance_invoices y finance_payments) para soportar nuevas funcionalidades. Se crearon 12 migraciones consecutivas para agregar columnas faltantes.

2.2 Error Detectado

Error P3018: A migration failed to apply
Error: P3018 A migration failed to apply. New migrations cannot be applied before the error is recovered from. Read more about how to resolve migration issues in a production database: https://pris.ly/d/migrate-resolve Migration name: 20250111_add_notes_columns Database error code: 42701 Database error: ERROR: column "notes" of relation "finance_invoices" already exists DbError { severity: "ERROR", code: SqlState(E42701), message: "column \"notes\" of relation \"finance_invoices\" already exists" }

2.3 Logs del Error en Producción

// Console del navegador mostrando el error GET https://legal-rag-api-qnew.onrender.com/api/v1/finance/invoices 500 (Internal Server Error) // Error de Prisma en logs del servidor { "type": "PrismaClientKnownRequestError", "message": "Invalid `prisma.invoiceFinance.findMany()` invocation:\n\nThe column `finance_invoices.internal_notes` does not exist in the current database.", "code": "P2022", "meta": { "modelName": "InvoiceFinance", "column": "finance_invoices.internal_notes" } }

2.4 Causa Raíz Identificada

Análisis de Causa Raíz: La columna notes ya existía en la tabla finance_invoices desde su creación inicial en la migración 20250111_calendar_tasks_notifications_finance

Hallazgos:

  • La tabla finance_invoices fue creada con la columna notes incluida
  • La migración 11 (20250111_add_notes_columns) intentó agregar nuevamente esta columna
  • Solo faltaba la columna internal_notes, no notes
  • La migración 12 también tenía redundancias con columnas ya existentes
-- Definición original de la tabla (migración inicial) CREATE TABLE "finance_invoices" ( "id" TEXT NOT NULL, "invoiceNumber" TEXT NOT NULL, ... "notes" TEXT, ← COLUMNA YA EXISTÍA AQUÍ "paymentTerms" TEXT, ... );

2.5 Complicaciones Adicionales

Pipeline Minutes Exhausted: El primer intento de fix falló porque la cuenta de Render había agotado los 500 minutos gratuitos mensuales de pipeline.

Problemas secundarios encontrados:

  1. Webhook no triggering: Auto-deploy no se activaba tras push a GitHub
  2. Pipeline Minutes: Cuenta sin minutos disponibles para builds
  3. Database Suspended: Base de datos PostgreSQL en modo sleep
  4. Read-Only Transactions: Imposibilidad de ejecutar ALTER TABLE via MCP tool

3. Planificación de la Solución

3.1 Estrategia de Resolución

Enfoque Adoptado: Eliminación de migraciones conflictivas y creación de una nueva migración idempotente con validaciones IF NOT EXISTS.

Decisiones de diseño:

  • ✅ Usar IF NOT EXISTS para evitar errores de duplicación
  • ✅ Eliminar migraciones problemáticas del repositorio
  • ✅ Actualizar script de resolución automática de migraciones
  • ✅ Marcar migración fallida como rolled-back en la base de datos
  • ❌ NO intentar reparar migraciones existentes (riesgo alto)
  • ❌ NO ejecutar ALTER TABLE directamente en producción

4. Implementación Técnica

4.1 Eliminación de Migraciones Conflictivas

Comando ejecutado:
rm -rf prisma/migrations/20250111_add_notes_columns
rm -rf prisma/migrations/20250111_add_remaining_metadata_notes

Migraciones eliminadas:

  • 20250111_add_notes_columns - Intentaba agregar notes (duplicada) + internal_notes
  • 20250111_add_remaining_metadata_notes - Tenía cambios redundantes

4.2 Nueva Migración: 20250111_fix_notes_columns

-- File: prisma/migrations/20250111_fix_notes_columns/migration.sql -- Fix: Add only the missing columns (notes already exists from table creation) -- Add internal_notes column to finance_invoices (this is the only missing column) ALTER TABLE "finance_invoices" ADD COLUMN IF NOT EXISTS "internal_notes" TEXT; -- Add metadata columns to both tables (if not exist) ALTER TABLE "finance_invoices" ADD COLUMN IF NOT EXISTS "metadata" JSONB; ALTER TABLE "finance_payments" ADD COLUMN IF NOT EXISTS "metadata" JSONB;
✅ Características clave de la solución:
  • Uso de IF NOT EXISTS previene errores de duplicación
  • Solo agrega columnas realmente faltantes
  • Idempotente - puede ejecutarse múltiples veces sin error
  • No modifica datos existentes

4.3 Script de Resolución Automática

7. Errores Encontrados y Resolución

7.1 Error Principal: Columna Duplicada

Error P3018: column "notes" of relation "finance_invoices" already exists

Causa: Intento de crear una columna que ya existía en la base de datos

Impacto: Deployment fallido, aplicación en producción con errores 500

Resolución: Eliminación de migración conflictiva y creación de nueva con IF NOT EXISTS

Tiempo de resolución: ~15 minutos

7.2 Error Secundario: Pipeline Minutes Exhausted

Render Pipeline: 500 minutos mensuales gratuitos agotados

Síntomas:

  • Webhook no triggering después de push
  • Builds no iniciándose automáticamente
  • Commits presentes en GitHub pero sin deployments

Resolución:

  • Usuario compró minutos adicionales de pipeline
  • Nuevo push trigger activó deployment exitosamente

7.3 Error Terciario: Database Suspended

PostgreSQL: Base de datos en modo sleep (tier gratuito)

Síntoma: Script wake-db-and-migrate.cjs falló después de 10 intentos

Impacto: Bajo - No crítico ya que deployment maneja el wake-up automáticamente

Resolución: No requirió acción - deployment manejó la conexión correctamente

7.4 Error Menor: Read-Only Transaction

MCP Tool: Cannot execute ALTER TABLE in a read-only transaction

Contexto: Intento de ejecutar migración directamente via MCP tool

Lección: Las herramientas MCP de consulta solo permiten operaciones SELECT

Resolución: Abandono de approach directo, uso de deployment para migraciones

9.2 Lecciones Aprendidas

⚠️ Lecciones importantes:
  1. Siempre usar IF NOT EXISTS en migraciones:

    Las migraciones deben ser idempotentes para evitar fallos por ejecuciones múltiples o columnas pre-existentes.

  2. Verificar historial completo de migraciones:

    Antes de crear una migración, revisar el historial completo para identificar si la columna ya fue creada previamente.

  3. Revisar definición inicial de tablas:

    Las tablas pueden haber sido creadas con columnas que el schema actual requiere, evitando la necesidad de migraciones adicionales.

  4. Implementar scripts de resolución automática:

    Scripts como resolve-failed-migrations.cjs son esenciales para manejar fallos de migración en producción.

  5. Monitorear pipeline minutes en servicios cloud:

    Los límites de recursos (como pipeline minutes) pueden bloquear deployments críticos.

  6. Mantener logs detallados:

    Logging comprehensivo facilitó la identificación rápida del problema y la validación de la solución.

9.3 Mejoras Implementadas

Cambios preventivos para el futuro:

  • ✅ Script de resolución actualizado para manejar nuevas migraciones fallidas
  • ✅ Uso de IF NOT EXISTS en todas las migraciones nuevas
  • ✅ Proceso documentado para debugging de migraciones
  • ✅ Validación mejorada antes de crear migraciones

9.4 Recomendaciones Futuras

📋 Recomendaciones para el equipo:
  1. Implementar testing de migraciones en staging:

    Crear un ambiente de staging que replique producción para probar migraciones antes de deployment.

  2. Automatizar validación de schema:

    Script de CI/CD que compare schema de Prisma con estado actual de base de datos.

  3. Mantener documentación de migraciones:

    Cada migración debe tener comentarios explicando qué agrega y por qué.

  4. Monitoreo proactivo de recursos:

    Alertas cuando pipeline minutes u otros recursos estén cerca de agotarse.

  5. Backup antes de migraciones críticas:

    Aunque esta migración era simple, migraciones complejas deben tener backup previo.

  6. Code review obligatorio para migraciones:

    Peer review de todas las migraciones antes de merge a main.
