Cutover totalmente aplicado. El backend Fastify ahora usa Supabase como Postgres principal, valida JWTs emitidos por Supabase Auth con custom claims (user_role, plan_tier), y todas las rutas autenticadas (/api/v1/*) operan contra el nuevo provider. 3Cronología del cutover

Punto 1 · JWT Custom Access Token Hook

Activado vía Supabase Management API. La función public.custom_access_token_hook inyecta user_role y plan_tier en el JWT leyéndolos de public.users. Esto habilita políticas RLS basadas en rol.

Punto 2 · OAuth Google

Creado proyecto Google Cloud powerai legal, OAuth client web con redirect https://lmnzzcqqegqugphcnmew.supabase.co/auth/v1/callback. Credenciales aplicadas en Supabase Auth via API. Verificado external_google_enabled = true.

Punto 3 · Import de usuarios Render → Supabase

2 usuarios importados con UUID id-match: el admin (benitocabrerar@gmail.com, bcrypt válido) y el test user (password fake → NULL para forzar magic link). Trigger handle_new_auth_user() creó row base en public.users; UPDATE posterior copió todos los campos legacy (role, plan_tier, avatar, etc.).

Cutover de DATABASE_URL

Reemplazado el connection string de Render por el pooler de Supabase. Render preservado en RENDER_DATABASE_URL. Auditoría previa: Render solo tenía 2 users y 1 sesión stale, sin corpus ni archivos productivos → no hay datos pendientes de migrar.

Frontend Next.js 15

Activado middleware.ts de Supabase, añadidos NEXT_PUBLIC_SUPABASE_URL y NEXT_PUBLIC_AUTH_BACKEND=supabase. Resueltos 4 errores de tipos pre-existentes (cookies API, SummaryMetadata.confidence, SummaryLanguage cast, conflicto auth/callback page vs route).

Redis Cloud reset

La DB original había sido borrada (la password rotó al inexistir el tenant). Creada DB nueva poweria-legal Free 30MB via Playwright contra Chrome con CDP. Endpoint y password extraídos automáticamente del modal Connect.

Verificación E2E del login

Reset password admin a Temporal2026! via Admin API. Test con Playwright contra el browser real: UI → signInWithPassword → cookies set → redirect a /dashboard → axios obtiene access_token de la sesión → backend valida via supabase.auth.getUser() → 200 GET /api/v1/cases.

4Trabas técnicas resueltas en vuelo

#SíntomaCausaResolución
1 Database error finding user en /auth/v1/otp Mi INSERT INTO auth.users dejó confirmation_token, recovery_token, etc. como NULL. Gotrue espera ''. UPDATE masivo a '' y patch del script de import para próximas corridas.
2 Tenant or user not found en Prisma vs Supabase URL del pooler con prefijo aws-0 y puerto 5432 en vez de aws-1:6543. Corregida URL al formato real del pooler con pgbouncer=true.
3 Cannot find package '@supabase/supabase-js' node_modules no tenía la dependencia instalada. npm install en backend y frontend (170 paquetes).
4 Build Next falla: dos pages resuelven al mismo path /auth/callback Coexistencia de page.tsx legacy con route.ts nuevo de Supabase OAuth. page.tsx renombrado a .legacy.
5 Build prerender falla: useAuth must be used within an AuthProvider SSG intentaba renderizar páginas client que dependen de contexto. export const dynamic = 'force-dynamic' en root layout.
6 Property 'get' does not exist on type 'Promise<ReadonlyRequestCookies>' Next 15 cambió cookies() a async. getSupabaseServerClient() ahora async con await cookies(); caller actualizado.
7 Login submit no llega a Supabase /login usaba fetch('/api/v1/auth/login') legacy en vez de useAuth().login(). Reescrito el handler para usar el contexto.
8 Layout dashboard expulsa al user a /login tras login OK SupabaseAuthProvider y AuthProvider creaban contextos separados; useAuth() leía el legacy (vacío). SupabaseAuthProvider ahora reusa el mismo AuthContext exportado de auth.tsx. Una sola línea de Provider, todos los consumidores intactos.
9 401 en GET /api/v1/cases tras login Axios interceptor leía localStorage('token') (vacío en modo Supabase). Interceptor async que hace supabase.auth.getSession() y usa el access_token cuando NEXT_PUBLIC_AUTH_BACKEND=supabase.
10 Backend rechaza JWT Supabase con 401 app.authenticate usaba request.jwtVerify() contra el secreto local. Decorator delega a requireSupabaseAuth cuando AUTH_BACKEND=supabase. Una sola edición → todas las rutas Fastify funcionan.
11 Error running hook URI: pg-functions://...custom_access_token_hook El role supabase_auth_admin no tenía EXECUTE ni SELECT sobre public.users. GRANT USAGE ON SCHEMA + GRANT EXECUTE ON FUNCTION + GRANT SELECT ON public.users TO supabase_auth_admin; revoke a authenticated/anon/public.
12 Redis Cloud rechaza login: WRONGPASS La instancia DB original había sido borrada (no rota la password — el tenant ya no existe). Creada nueva instancia free via Playwright/CDP. Endpoint y password extraídos del modal "Connect".
13 bcrypt del admin no coincide con passwords conocidas El hash en Render era de una password olvidada / ya rotada. Reset via Supabase Admin API a Temporal2026!.

7Pendientes y próximos pasos

Password temporal: el admin tiene Temporal2026! tras el reset via Admin API. Debe cambiarse desde la UI o el Dashboard de Supabase antes de cualquier uso compartido.

Prioridad alta (bloquea uso pleno)

  • Cambiar la password temporal del admin (UI /account/password o Dashboard Supabase → Authentication → Users → Edit user)
  • Verificar el resto de rutas autenticadas (/api/v1/documents, /api/v1/ai-assistant, /api/v1/legal-documents, etc.) con el JWT nuevo. El decorator app.authenticate es global, así que deberían funcionar directamente, pero merece smoke-test.

Prioridad media (mejoras de UX y seguridad)

  • Botón "Iniciar con Google" en /login que invoque loginWithGoogle() exportado en lib/auth-supabase.tsx. El provider ya está configurado en Supabase.
  • Configurar SMTP en Supabase (Authentication → Email Settings) para que los flows de magic link, recuperación de password y signup funcionen en producción. Hoy solo funcionan via Admin API.
  • UI de signup que use useAuth().register() (ya cableado al contexto). Validar que el trigger handle_new_auth_user() crea row en public.users con plan free.
  • Limpiar page.tsx.legacy de frontend/src/app/auth/callback/ cuando esté confirmado que no se necesita.
  • Revisar Redis Cloud — alerta cosmética: eviction policy es volatile-lru, ideal sería noeviction para que rate-limit no se pierda bajo presión de memoria.

Prioridad baja (deuda diferida)

  • Re-embedding del corpus jurídico cuando se ingesten documentos (hoy hay 0 en legal_documents y document_chunks). Costo estimado: ~$1 por cada 100k chunks vía OpenAI.
  • Migrar archivos S3 a Supabase Storage cuando existan (hoy no hay nada productivo en S3). Adapter dual-write ya disponible vía STORAGE_DUAL_WRITE_LEGACY=true.
  • Borrar la subscripción Render Postgres cuando hayan pasado 14 días sin necesidad de rollback.
  • Promover el frontend a producción (Vercel) con las nuevas envs NEXT_PUBLIC_SUPABASE_* y NEXT_PUBLIC_API_URL apuntando al nuevo backend.
  • Migrar BullMQ workers a pgmq nativo de Supabase (Fase 4 del plan original) — no urgente; BullMQ funciona con Redis nuevo.
  • Eliminar dependencia pinecone de package.json — no hay imports reales en el código.
  • 2FA: tabla custom (two_factor_* en public.users) vs migrar a auth.mfa_factors de Supabase. Hoy nadie tiene 2FA habilitado, así que la decisión puede esperar.

8Procedimiento de rollback (~1 min)

Cuándo aplicarlo: sólo si Supabase se cae durante una ventana crítica y Render sigue operacional. Render tenía 2 users y 1 sesión stale, así que hay paridad de datos.
  1. En backend/.env: comentar AUTH_BACKEND, SEARCH_BACKEND, STORAGE_BACKEND.
  2. En backend/.env: intercambiar DATABASE_URL ↔ RENDER_DATABASE_URL.
  3. En frontend/.env.local: comentar NEXT_PUBLIC_AUTH_BACKEND, apuntar NEXT_PUBLIC_API_URL al backend Render anterior.
  4. En frontend/: mv middleware.ts middleware.ts.example.
  5. Restart de backend y frontend. Verificar login en /login.

Después de 14 días sin necesidad de rollback, eliminar RENDER_DATABASE_URL y dar de baja la instancia Render para no seguir pagando.
