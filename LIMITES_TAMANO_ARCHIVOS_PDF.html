📊 Límites Actuales de Tamaño

Límite Principal Configurado

Tamaño máximo de archivo PDF: 50 MB

Este límite se aplica cuando los usuarios suben documentos legales a través del panel de administración.

Ubicaciones de los Límites

El límite de 50MB está configurado en tres capas del sistema para máxima seguridad:

Capa 1 Frontend (Validación Cliente)

frontend/src/app/admin/legal-library/page.tsx
if (file.size > 50 * 1024 * 1024) {
  alert('El archivo no debe superar 50MB');
  return;
}

Propósito: Validación inmediata antes de enviar al servidor, mejora UX.

Capa 2 Backend Multer (Middleware de Upload)

src/routes/documents-enhanced.ts
const upload = multer({
  storage,
  limits: {
    fileSize: 50 * 1024 * 1024, // 50MB max
  },
  fileFilter: (req, file, cb) => {
    const allowedTypes = [
      'application/pdf',
      'application/msword',
      'application/vnd.openxmlformats-officedocument.wordprocessingml.document',
      'text/plain',
      'image/jpeg',
      'image/png'
    ];

    if (allowedTypes.includes(file.mimetype)) {
      cb(null, true);
    } else {
      cb(new Error('Invalid file type. Only PDF, DOC, DOCX, TXT, JPG, and PNG files are allowed.'));
    }
  }
});

Propósito: Protección del servidor contra uploads excesivos.

Capa 3 Fastify Multipart (Configuración Global)

src/server.ts
await app.register(multipart);

Límite por defecto de @fastify/multipart: Sin límite explícito

⚠️ Fastify no tiene límite de body configurado explícitamente en el servidor principal.

⚙️ Cómo Modificar los Límites

⚠️ ADVERTENCIA IMPORTANTE

Modificar los límites de tamaño de archivos afecta:

  • Rendimiento del servidor: Archivos más grandes consumen más memoria RAM
  • Tiempo de procesamiento: Vectorización de archivos grandes es intensiva en CPU
  • Costos de API: OpenAI cobra por tokens (más texto = más costo)
  • Experiencia del usuario: Uploads largos pueden causar timeouts
  • Almacenamiento: AWS S3 cobra por GB almacenado

const maxFileSizeBytes = maxFileSizeMB * 1024 * 1024;

await app.register(multipart, {

⚖️ Consideraciones Técnicas

1. Impacto en Rendimiento

Tamaño de Archivo Memoria RAM Requerida Tiempo de Procesamiento Costo OpenAI (Aprox.)
10 MB (PDF) ~30-50 MB RAM 5-10 segundos $0.05 - $0.15
50 MB (PDF) ~150-250 MB RAM 30-60 segundos $0.25 - $0.75
100 MB (PDF) ~300-500 MB RAM 60-120 segundos $0.50 - $1.50
200 MB (PDF) ~600 MB - 1 GB RAM 120-240 segundos $1.00 - $3.00

⚠️ Límites de Render (Free Tier)

  • RAM: 256 MB
  • Timeout: 60 segundos por request
  • Conexiones concurrentes: Limitadas

Con el plan gratuito de Render, archivos de más de 50MB pueden causar:

  • ❌ Out of Memory (OOM) errors
  • ❌ Request timeouts
  • ❌ Servidor colgado/reiniciado

2. Impacto en Costos

OpenAI API (text-embedding-ada-002)

  • Costo: $0.0001 por 1,000 tokens
  • Tokens por página: ~500-800 tokens promedio
  • PDF de 50MB: ~500-1000 páginas = 250,000 - 800,000 tokens
  • Costo estimado 50MB: $0.025 - $0.08 por documento

AWS S3 (Almacenamiento)

  • Costo: $0.023 por GB/mes (US-East)
  • 100 documentos de 50MB: 5 GB = $0.115/mes
  • 1000 documentos de 50MB: 50 GB = $1.15/mes

Render PostgreSQL

  • Free Tier: 256 MB RAM, 1 GB storage
  • Embeddings por documento 50MB: ~500-1000 vectores × 6KB = 3-6 MB
  • Límite aprox: 150-300 documentos de 50MB antes de necesitar upgrade

3. Límites de Terceros

Servicio Límite Configurable
Render (Free Tier) 256 MB RAM, 60s timeout No (upgrade requerido)
OpenAI API 8,191 tokens por request (ada-002) No (límite del modelo)
AWS S3 5 TB por objeto (PUT), sin límite total Sí (prácticamente ilimitado)
Cloudinary (si se usa) 10 MB en Free tier Limitado (upgrade requerido)
PostgreSQL 1 GB en Render Free Limitado (upgrade requerido)

✅ Mejores Prácticas

1. Validación en Múltiples Capas

Implementar validaciones redundantes para máxima seguridad:

  • ✅ Cliente (JavaScript): Feedback inmediato, mejor UX
  • ✅ Servidor (Middleware): Protección contra bypasseo de frontend
  • ✅ Base de datos (Constraints): Última línea de defensa

2. Monitoreo de Recursos

Implementar métricas para detectar problemas:

// Logging de tamaño de archivos
fastify.log.info({
  event: 'file_uploaded',
  fileSize: file.size,
  fileSizeMB: (file.size / (1024 * 1024)).toFixed(2),
  filename: file.filename,
  userId: user.id
});

// Alertas si se acerca al límite de RAM
if (process.memoryUsage().heapUsed > 200 * 1024 * 1024) {
  fastify.log.warn('High memory usage detected');
}

4. Límites Progresivos por Plan

Diferentes límites según el tipo de usuario:

const fileSizeLimits = {
  free: 10 * 1024 * 1024,        // 10MB
  basic: 25 * 1024 * 1024,       // 25MB
  professional: 50 * 1024 * 1024, // 50MB
  enterprise: 200 * 1024 * 1024   // 200MB
};

const userPlan = await getUserPlan(user.id);
const maxSize = fileSizeLimits[userPlan] || fileSizeLimits.free;

if (file.size > maxSize) {
  return reply.code(413).send({
    error: 'Payload Too Large',
    message: `Su plan ${userPlan} permite archivos de hasta ${maxSize / (1024 * 1024)}MB`,
    upgradeUrl: '/subscription/upgrade'
  });
}

5. Compresión y Optimización

Optimizar PDFs antes de procesar:

  • Remover imágenes innecesarias: Extraer solo texto
  • Comprimir imágenes embebidas: Reducir calidad a 72 DPI
  • Eliminar metadata: Remover propiedades no esenciales
  • OCR solo cuando necesario: Detectar si es PDF de imagen

6. Caché de Embeddings

Evitar reprocesar documentos idénticos:

// Generar hash del contenido
const contentHash = crypto
  .createHash('sha256')
  .update(fileBuffer)
  .digest('hex');

// Buscar si ya existe
const existingDocument = await prisma.legalDocument.findFirst({
  where: { contentHash }
});

if (existingDocument) {
  return reply.send({
    message: 'Este documento ya existe en el sistema',
    documentId: existingDocument.id,
    duplicate: true
  });
}

7. Chunking Inteligente

Dividir documentos grandes en fragmentos óptimos:

const CHUNK_SIZE = 1000; // caracteres
const CHUNK_OVERLAP = 200; // solapamiento entre chunks

function smartChunk(text: string): string[] {
  const chunks = [];
  let start = 0;

  while (start < text.length) {
    let end = start + CHUNK_SIZE;

    // Buscar el último punto antes del límite
    if (end < text.length) {
      const lastPeriod = text.lastIndexOf('.', end);
      if (lastPeriod > start) {
        end = lastPeriod + 1;
      }
    }

    chunks.push(text.slice(start, end));
    start = end - CHUNK_OVERLAP; // Overlap para contexto
  }

  return chunks;
}

🔧 Solución de Problemas Comunes

Problema 1: Error "Payload Too Large"

Síntoma

413 Payload Too Large o Request entity too large

Causa

El archivo excede el límite configurado en el servidor.

Solución

  1. Verificar límite en src/routes/documents-enhanced.ts
  2. Verificar límite en src/server.ts (multipart config)
  3. Aumentar límites según método 3 o 4 en sección "Cómo Modificar"
  4. Reiniciar servidor después de cambios

Problema 2: Timeout durante Upload

Síntoma

Request se cancela después de ~60 segundos, especialmente con archivos grandes.

Causa

El procesamiento del PDF toma más tiempo que el timeout configurado.

Solución

// En server.ts
const app = Fastify({
  logger: true,
  requestTimeout: 300000,    // 5 minutos
  connectionTimeout: 300000  // 5 minutos
});

// O implementar procesamiento asíncrono (recomendado)

Problema 3: Out of Memory (OOM)

Síntoma

Servidor se reinicia o crashea al procesar archivos grandes. Error: JavaScript heap out of memory

Causa

El archivo completo se carga en memoria RAM, excediendo los límites de Render Free (256MB).

Solución

  1. Corto plazo: Reducir límite de archivo a 25-30MB
  2. Medio plazo: Usar procesamiento por streams en lugar de buffers
  3. Largo plazo: Upgrade a plan de Render con más RAM (Starter: 512MB, $7/mes)
// Procesamiento por streams (reduce uso de memoria)
import { pipeline } from 'stream/promises';

const readStream = fs.createReadStream(filePath);
const pdfParser = new PDFParser();

await pipeline(
  readStream,
  pdfParser,
  processChunks // Procesar de a poco
);

Problema 4: Frontend no Valida Correctamente

Síntoma

El frontend permite subir archivos que exceden el límite del backend.

Causa

Límite en frontend no coincide con límite en backend.

Solución

Sincronizar límites usando variable de entorno compartida:

// .env (backend)
MAX_FILE_SIZE_MB=50

// frontend/.env.local
NEXT_PUBLIC_MAX_FILE_SIZE_MB=50

// Validación consistente en ambos lados

Problema 5: Costos Elevados de OpenAI

Síntoma

Factura de OpenAI más alta de lo esperado.

Causa

Procesar documentos grandes genera muchos embeddings (muchos tokens).

Solución

  1. Implementar caché: No reprocesar documentos duplicados
  2. Optimizar chunks: Reducir solapamiento entre fragmentos
  3. Filtrar texto: Remover encabezados/pies de página repetitivos
  4. Límite inteligente: Cobrar a usuarios enterprise por documentos >50MB
// Estimación de costo antes de procesar
const estimatedTokens = Math.ceil(textLength / 4);
const estimatedCost = (estimatedTokens / 1000) * 0.0001;

if (estimatedCost > 1.0 && userPlan !== 'enterprise') {
  return reply.code(402).send({
    error: 'Payment Required',
    message: `Este documento generaría un costo estimado de $${estimatedCost.toFixed(2)}.
               Solo usuarios Enterprise pueden procesar documentos de este tamaño.`,
    upgradeUrl: '/subscription/upgrade'
  });
}

Problema 6: Almacenamiento S3 Lleno

Síntoma

Errores al subir nuevos documentos o costos elevados de S3.

Causa

Acumulación de archivos sin política de limpieza.

Solución

  1. Lifecycle policies: Eliminar archivos >90 días automáticamente
  2. Versioning: Mantener solo última versión
  3. Intelligent-Tiering: Mover archivos viejos a storage más barato

🎯 Recomendaciones Finales

Para Entornos de Producción

  • ✅ Límite recomendado: 50MB (balanceo entre usabilidad y recursos)
  • ✅ Usar variables de entorno: Facilita cambios sin recompilar
  • ✅ Implementar monitoreo: Alertas cuando archivos grandes se procesan
  • ✅ Procesamiento asíncrono: Usar BullMQ para archivos >25MB
  • ✅ Upgrade Render: Mínimo Starter plan ($7/mes) para 512MB RAM

Para Entornos de Desarrollo/Testing

  • 💡 Límite relajado: 100MB para probar casos extremos
  • 💡 Logs verbosos: Registrar todo el proceso de upload
  • 💡 Archivos de prueba: Generar PDFs de diferentes tamaños para testing
