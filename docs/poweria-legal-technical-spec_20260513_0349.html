01Resumen ejecutivo

Poweria Legal es una plataforma SaaS B2B/B2C diseñada para abogados litigantes y firmas legales en Ecuador. Combina gestión documental forense, un asistente conversacional con IA entrenado contextualmente con cada expediente, generación asistida de escritos legales con citas verificables y análisis profundo basado en RAG sobre el corpus normativo nacional vectorizado.

Flujo típico — Subida de documento

  1. Drag-and-drop en CaseAIChat o DocumentUploadProgress.
  2. POST multipart /api/v1/documents/upload → Fastify recibe, extrae texto (pdf-parse, Tesseract Vision, mammoth, xlsx).
  3. Insert en documents con kind='uploaded' y trigger set_country_code.
  4. Upload binario a Supabase Storage bucket legal-documents.
  5. Chunking 1000 chars con overlap 200 → embeddings OpenAI batch 50 → INSERT en document_chunks (vector + FTS).
  6. SSE brain.refresh regenera el "cerebro" del caso (síntesis multi-doc).
  7. Cliente abre PostUploadAnalysisDialog → SSE a /cases/:id/documents/:docId/post-upload-analysis con Claude Opus 4.7.
  8. Análisis estructurado JSON → persistido como kind='ai_analysis' linkeado al original.

ErrorBoundary.tsx (auto-recover)

Detecta ChunkLoadError tras un deploy. Cuando ocurre: purga toda la caches entries + unregister de todos los SW + window.location.reload(). sessionStorage flag con timestamp evita loop infinito.

tsx
// ErrorBoundary.tsx — detección + recovery
const isChunkError =
  error?.name === 'ChunkLoadError' ||
  /Loading chunk \d+ failed/i.test(error?.message);

if (isChunkError) {
  const last = sessionStorage.getItem(RELOAD_FLAG);
  if (!last || Date.now() - Number(last) > 30000) {
    await Promise.all((await caches.keys()).map(k => caches.delete(k)));
    const regs = await navigator.serviceWorker.getRegistrations();
    await Promise.all(regs.map(r => r.unregister()));
    window.location.reload();
  }
}

6.2 Trigger de reemplazo

Cuando se inserta un Document con replaces_document_id != null, un trigger setea replaced_at = now() en el documento original. Esto preserva la historia sin perder relaciones:

sql
CREATE OR REPLACE FUNCTION mark_replaced_document() RETURNS TRIGGER AS $$
BEGIN
  IF NEW.replaces_document_id IS NOT NULL THEN
    UPDATE documents
    SET replaced_at = now()
    WHERE id = NEW.replaces_document_id;
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

08Pipeline de IA

8.1 Modelo principal

Claude Opus 4.7 es el modelo de razonamiento default. Contexto de 1M tokens permite cargar expedientes completos (cientos de páginas) sin truncar. Características operacionales:

  • No acepta parámetro temperature (omitido por NO_TEMPERATURE_MODELS set).
  • Streaming nativo via Anthropic SSE → traducido a chunks OpenAI-shape por el wrapper interno.
  • Max tokens default por endpoint: 4000 (analysis) — 8000 (prompts generators).
  • System prompts versionados en cada route; usuarios siempre españoles ecuatorianos profesionales.

8.2 Anti-alucinación

El sistema implementa varias capas para evitar invención de citas:

  1. RAG obligatorio en generación de documentos: el prompt incluye los chunks recuperados como "MARCO NORMATIVO" y se instruye al modelo a citar solo de allí.
  2. Marcador [CITA POR VERIFICAR]: si el modelo no tiene certeza absoluta sobre un artículo, debe escribir literalmente ese marcador en lugar de inventar.
  3. Estado de los documentos: en los prompts de análisis se indica explícitamente el kind de cada documento del expediente para que la IA priorice court_filed sobre borradores.

8.3 Parser tolerante a truncamiento

Cuando el modelo devuelve JSON y se corta por max_tokens, en lugar de fallar usamos repairTruncatedPromptsJson() que escanea carácter a carácter respetando strings/escapes, encuentra el último } cerrado dentro del array prompts y rearma manualmente cerrando ]}:

ts
function repairTruncatedPromptsJson(s: string): any | null {
  let depthBrace = 0, inString = false, escape = false;
  let lastSafePromptEnd = -1;
  let arrayStarted = false;
  for (let i = 0; i < s.length; i++) {
    const ch = s[i];
    if (escape) { escape = false; continue; }
    if (ch === '\\') { escape = true; continue; }
    if (ch === '"') { inString = !inString; continue; }
    if (inString) continue;
    if (ch === '{') depthBrace++;
    else if (ch === '}') {
      depthBrace--;
      if (arrayStarted && depthBrace === 1) lastSafePromptEnd = i;
    }
    else if (ch === '[' && !arrayStarted) arrayStarted = true;
  }
  if (lastSafePromptEnd < 0) return null;
  return JSON.parse(s.slice(0, lastSafePromptEnd + 1) + ']}');
}

9.1 Corpus legal vectorizado

138 cuerpos normativos ecuatorianos pre-procesados:

  • Código Orgánico Integral Penal (COIP)
  • Código Orgánico General de Procesos (COGEP)

  • Ley Orgánica de Garantías Jurisdiccionales, Ley de Compañías, etc.
  • Reglamentos, resoluciones de la Corte Constitucional y Corte Nacional

9.2 Pipeline de chunking

  1. Extracción: PDFs procesados con pdf-parse + Vision IA para escaneos.
  2. Chunking: ~1000 chars con overlap 200, respeta límites de oración cuando es posible.
  3. Embedding: OpenAI text-embedding-3-small, 1536 dimensiones, batch size 50.
  4. Indexación: pgvector con índice HNSW (m=16, ef_construction=64) + tsvector spanish para BM25.
  5. Metadata: norm_type, jurisdiction, country_code, article_number, section.

9.3 Búsqueda híbrida (RRF)

El RPC search_legal_chunks(query_embedding, query_text, match_count, semantic_weight, keyword_weight, filter_*) ejecuta dos consultas en paralelo y las fusiona con Reciprocal Rank Fusion:

sql
WITH semantic AS (
  SELECT id, embedding <=> query_embedding AS distance
  FROM legal_document_chunks
  ORDER BY distance LIMIT 30
), keyword AS (
  SELECT id, ts_rank(ts, plainto_tsquery('spanish', query_text)) AS rank
  FROM legal_document_chunks
  WHERE ts @@ plainto_tsquery('spanish', query_text)
  ORDER BY rank DESC LIMIT 30
)
SELECT id,
  (semantic_weight / (60 + s.row_num) + keyword_weight / (60 + k.row_num)) AS rrf_score
FROM semantic s FULL JOIN keyword k USING (id)
ORDER BY rrf_score DESC LIMIT match_count;

11Server-Sent Events

Todos los endpoints de IA del producto usan SSE para entregar progreso en tiempo real. La implementación está endurecida contra los problemas típicos de proxies (Render + Cloudflare):

11.1 Helper setSseHeaders

ts
export function setSseHeaders(req: FastifyRequest, reply: FastifyReply): void {
  const origin = req.headers.origin;
  if (origin && isOriginAllowed(origin)) {
    reply.raw.setHeader('Access-Control-Allow-Origin', origin);
    reply.raw.setHeader('Access-Control-Allow-Credentials', 'true');
    reply.raw.setHeader('Vary', 'Origin');
  }
  reply.raw.setHeader('Content-Type', 'text/event-stream');
  reply.raw.setHeader('Cache-Control', 'no-cache, no-transform');
  reply.raw.setHeader('Connection', 'keep-alive');
  reply.raw.setHeader('X-Accel-Buffering', 'no');
  reply.raw.flushHeaders?.();

  // 1) Disable Nagle — TCP_NODELAY para chunks pequeños
  (reply.raw.socket as any)?.setNoDelay?.(true);

  // 2) Preámbulo 4KB para forzar a proxies a desbufferar
  reply.raw.write(`: ${'#'.repeat(4096)}\n\n`);
}

11.2 Keepalive 1s

Para evitar idle timeout y "stalled" del browser durante operaciones largas (RAG + IA puede tardar 30-60s), todos los endpoints SSE arrancan un intervalo:

ts
export function startSseKeepalive(reply: FastifyReply, ms = 1000): () => void {
  const handle = setInterval(() => {
    try { reply.raw.write(`: ka ${Date.now()}\n\n`); }
    catch { clearInterval(handle); }
  }, ms);
  return () => clearInterval(handle);
}
Lección aprendida
TCP_NODELAY + preámbulo + keepalive son requisito obligatorio para SSE en Render. Sin esto, el cliente ve la barra de progreso colgada en 0% indefinidamente porque los proxies acumulan 4KB antes de iniciar la entrega.

11.3 Cliente — parser SSE robusto

El frontend filtra líneas de comentario : (no-op por spec), implementa fallback content-type (si no es text/event-stream parsea como JSON legacy) y watchdog 90s de inactividad:

ts
while (true) {
  const { done, value } = await reader.read();
  if (done) break;
  bumpActivity();
  buffer += decoder.decode(value, { stream: true });
  let idx;
  while ((idx = buffer.indexOf('\n\n')) >= 0) {
    const chunk = buffer.slice(0, idx);
    buffer = buffer.slice(idx + 2);
    // Ignorar comentarios SSE (preámbulo keepalive)
    const lines = chunk.split('\n').filter((l) => !l.startsWith(':'));
    const evLine = lines.find((l) => l.startsWith('event:'));
    const dataLine = lines.find((l) => l.startsWith('data:'));
    if (!dataLine) continue;
    const event = evLine?.slice(6).trim() ?? 'message';
    const payload = JSON.parse(dataLine.slice(5).trim());
    handle(event, payload);
  }
}

17Seguridad y compliance

17.1 Cifrado

  • TLS 1.3 en todas las conexiones (Vercel + Render + Supabase tienen HTTPS-only).
  • At-rest: Supabase Postgres AES-256, Storage S3 SSE-S3.
  • Application-level: campos sensibles (tokens, claves API) cifrados con field-encryption.service.

17.2 LOPDP Ecuador

Sección /lopdp declara:

  • Datos del cliente NUNCA entrenan modelos de IA (cláusula contractual con Anthropic/OpenAI).
  • Derecho de acceso, rectificación, cancelación y oposición (ARCO).
  • DPO (Data Protection Officer) designado en COGNITEX.
  • Plazos de retención: documentos del caso mantenidos por la duración de la suscripción + 60 días gracia.
  • Notificación de brechas en 72h.

17.3 Defensas técnicas

  • Helmet headers (X-Content-Type-Options, Referrer-Policy, Permissions-Policy).
  • Validación Zod en todo input externo.
  • Rate limiting Redis-backed.
  • SQL injection prevenido por Prisma + parameterized queries.
  • XSS: React escape por defecto + sanitización de markdown.
  • CSRF: Bearer tokens + SameSite cookies.
