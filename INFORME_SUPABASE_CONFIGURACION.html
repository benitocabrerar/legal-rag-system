4.3 Trade-offs de la Decisión

Por qué Render + Prisma fue elegido:

  • ✅ Simplicidad: Stack unificado (Prisma para todo)
  • ✅ Type-safety: Prisma genera tipos TypeScript automáticamente
  • ✅ Portabilidad: Código no acoplado a proveedor específico
  • ✅ Control de versiones: Migraciones con Prisma Migrate en Git
  • ✅ Debugging: Queries SQL visibles y modificables

Por qué Supabase no fue elegido:

  • ❌ Vendor lock-in: APIs propietarias de Supabase
  • ❌ Complejidad adicional: Cliente Supabase + Prisma sería redundante
  • ❌ Costo: Free tier de Supabase más limitado (500MB)
  • ❌ Overhead: Características de Supabase (Auth, Storage, Real-time) no necesarias inicialmente
