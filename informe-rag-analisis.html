Resumen Ejecutivo Lectura: 90 segundos

Cinco hallazgos no negociables

El sistema RAG actual presenta defectos arquitectónicos que invalidan la suposición de "95% completo" declarada en el README: el motor de búsqueda no es realmente semántico a escala, el aislamiento multi‑tenant depende de disciplina manual, y existen tres rutas de retrieval coexistentes y mutuamente inconsistentes.

  1. 1
    Embeddings sin pgvector real

    La columna embedding es jsonb, no vector(1536). La extensión y los índices IVFFlat están comentados en migraciones. Cosine similarity se calcula en memoria JS sobre >1.000 chunks por query.

  2. 2
    Tres pipelines de retrieval coexistiendo

    routes/query.ts (brute‑force JS), unified-search-orchestrator.ts (ILIKE + rerank) y queryRouter.ts (SQL con <=> y FTS) usan modelos y semánticas distintas.

  3. 3
    Multi‑tenant sin RLS

    Cero CREATE POLICY y cero ENABLE ROW LEVEL SECURITY en todo el repo. El aislamiento depende de un where: { userId } en cada handler — un único olvido convierte un endpoint en un IDOR.

  4. 4
    Chunking naïve + modelos inconsistentes

    La ingestión usa fixed‑size 1.000 chars sin overlap; el hierarchicalChunker.ts existe pero no se invoca. Conviven ada-002 y text-embedding-3-small en distintas rutas.

  5. 5
    Veredicto: B · Migrar a Supabase por fases

    Supabase no introduce un cambio de paradigma — es PostgreSQL nativo con pgvector, RLS, Auth, Storage, Realtime y Vault gestionados. La migración resuelve simultáneamente los cuatro hallazgos anteriores con esfuerzo estimado 120–180 h, riesgo medio y ahorro neto en TCO desde el primer mes a partir de 100k chunks. El lock‑in real es bajo: el SQL es portable y los embeddings se exportan como CSV.

FASE 2

Arquitectura equivalente en Supabase

El rediseño no cambia el modelo de datos: aprovecha que Supabase es PostgreSQL gestionado y reemplaza componentes que actualmente se escriben a mano (Auth, Storage, Vault, RLS, branching, edge runtime) por equivalentes nativos. Las extensiones vector, pg_trgm, pgcrypto, uuid-ossp vienen pre‑instaladas y administradas.

2.1 · Esquema corregido con pgvector nativo

-- Extensiones gestionadas por Supabase (un click en Dashboard)
create extension if not exists vector;
create extension if not exists pg_trgm;
create extension if not exists pgcrypto;

-- Embedding nativo con índice HNSW (recomendado para < 10M vectores)
alter table legal_document_chunks
  add column embedding vector(1536);

create index idx_ldc_embedding_hnsw
  on legal_document_chunks
  using hnsw (embedding vector_cosine_ops)
  with (m = 16, ef_construction = 64);

-- FTS y trigram para fallback léxico (hybrid search)
create index idx_ldc_content_fts
  on legal_document_chunks
  using gin (to_tsvector('spanish', content));

-- RLS multi-tenant (lo que hoy es manual con `where userId`)
alter table documents enable row level security;
create policy "own documents" on documents
  for all to authenticated
  using  (user_id = (select auth.uid()))
  with check (user_id = (select auth.uid()));

-- Catálogo legal compartido: lectura pública, escritura solo admin
alter table legal_documents enable row level security;
create policy "public read legal" on legal_documents
  for select using (is_active);
create policy "admin write" on legal_documents
  for insert with check (
    (select auth.jwt() ->> 'role') = 'admin'
  );

2.3 · Pipeline de retrieval unificado (un solo path)

-- Hybrid search con RRF (Reciprocal Rank Fusion) en 1 query
with sem as (
  select id, document_id,
         row_number() over (order by embedding <=> $1) as r
  from legal_document_chunks
  where $2::uuid is null or document_id = $2
  order by embedding <=> $1
  limit 60
), kw as (
  select id, document_id,
         row_number() over (
           order by ts_rank_cd(
             to_tsvector('spanish', content),
             websearch_to_tsquery('spanish', $3)
           ) desc
         ) as r
  from legal_document_chunks
  where to_tsvector('spanish', content) @@ websearch_to_tsquery('spanish', $3)
  limit 60
)
select coalesce(sem.id, kw.id) as id,
       1.0/(60 + coalesce(sem.r, 9999)) +
       1.0/(60 + coalesce(kw.r,  9999)) as rrf
from sem full outer join kw using (id)
order by rrf desc
limit 20;

3.3 · Pros y contras por alternativa

Mantener stack actual
Pros
  • Cero migración, cero riesgo inmediato.
  • Postgres puro, control total del runtime.
  • El equipo ya conoce el repo.
Contras
  • Tres pipelines de retrieval inconsistentes — uno está roto en runtime.
  • Embeddings en jsonb = sin índice vectorial real.
  • Cero RLS — un único findMany mal escrito = brecha.
  • ~6.500 LOC de infra reescrita a mano (auth, cache, queue).
  • Costo creciente con datos por full‑scan.
Migrar a Supabase
Pros
  • pgvector con HNSW gestionado, FTS spanish OOTB.
  • RLS forzado a nivel motor — defensa en profundidad real.
  • Auth, Storage, Realtime, Vault, branching incluidos.
  • SDK first‑class para Next.js + Server Components.
  • Schema 100% portable (es PostgreSQL).
Contras
  • Migración inicial 120–180 h con plan estructurado.
  • Edge Functions = Deno (cambio leve para tareas asíncronas).
  • Pricing por compute size; subir size requiere planificación.
  • Realtime tiene límites de conexiones por plan.

FASE 4

Veredicto técnico

B
Recomendación inequívoca

Migrar integralmente a Supabase, en 4 fases, en 6–9 semanas

El stack actual no es un sistema RAG funcional a escala: es un sistema RAG en un estado intermedio donde la pieza más cara (la base vectorial) está implementada como búsqueda lineal en memoria sobre jsonb. La migración a Supabase no aporta una mejora marginal — corrige el problema central, cierra el riesgo de seguridad multi‑tenant y reduce ~6.500 LOC de infraestructura escrita a mano. El lock‑in es bajo por construcción: Supabase es Postgres + extensiones documentadas.

Criterios objetivos de éxito (SLO post‑migración)
  • Latencia retrieval p95 < 250 ms a 100k chunks.
  • Latencia retrieval p99 < 600 ms a 1M chunks.
  • Recall@10 ≥ 0,90 en evaluación interna de 200 queries.
  • Zero findMany sin filtro userId — tests RLS en CI.
  • Embedding único: text-embedding-3-small en toda la base.
  • 0 rutas con cosine en JS — todo el cómputo en Postgres.
  • Costo mensual ≤ 70% del baseline a igual volumen.
  • RTO < 1 h, RPO < 5 min con backups de Supabase.

Fase 0 · Preparación · 8 h
  • Crear org y proyecto Supabase Pro, región más cercana a usuarios.
  • Configurar branching y un branch migracion.
  • Snapshot de la BD productiva (pg_dump --no-owner).
  • Definir golden set de 200 queries para evaluación recall.
Fase 1 · Datos y schema · 30 h
  • Restaurar dump en Supabase, ajustar tipos (uuid, timestamps tz).
  • ALTER TABLE … ADD COLUMN embedding vector(1536).
  • Job de re‑embedding por lotes (10k chunks/hora) con text-embedding-3-small.
  • Crear índices HNSW y GIN, verificar EXPLAIN ANALYZE.
  • Eliminar columna jsonb antigua tras backfill.
Fase 2 · RLS + Auth · 30 h
  • Inventario de tablas con datos de usuario (Cases, Documents, Tasks, etc.).
  • Una policy por tabla, basadas en auth.uid() y auth.jwt() ->> 'role'.
  • Migrar tabla users a auth.users con raw_app_meta_data para roles.
  • Endpoint de bootstrap que envía magic link masivo a usuarios existentes.
  • Tests Vitest que verifican que un usuario A no puede ver entidades de B.
Fase 3 · Aplicación · 60 h
  • Reemplazar routes/query.ts, unified-search-orchestrator.ts y queryRouter.ts por una sola RPC search_legal() en Postgres con RRF.
  • Migrar uploads a Supabase Storage con políticas idénticas a RLS de tablas.
  • Edge Functions Deno: ingest-document (chunking + embed), summarize-stream.
  • Frontend: cliente @supabase/ssr en server components, @supabase/supabase-js en client.
  • Borrar Pinecone deps + multi‑tier cache custom no necesario.
Fase 4 · Cutover y monitoreo · 25 h
  • Dual‑write 48 h: backend Render escribe a ambas BD, lecturas siguen en Render.
  • Canary 5 % de tráfico a Supabase, métricas latencia y error rate en paralelo.
  • Switch DNS / variable DATABASE_URL → Supabase.
  • Mantener Render como hot standby por 14 días.
  • Decommission de Pinecone, S3‑avatares, Redis Render Starter (consolidados en Supabase).
