8Pendientes (deuda diferida)

Funcional · cuando lo necesites

  • Configurar SMTP propio en Supabase. Hoy mailer_autoconfirm=true para evitar el rate-limit del SMTP shared. Una vez tengas SMTP propio (SendGrid ya en uso, Postmark, AWS SES), desactiva autoconfirm para que signup envíe email de verificación, magic link funcione, y password recovery por email esté disponible. Lugar: Supabase Dashboard → Authentication → Email Settings.
  • Re-embedding del corpus. Ejecutar npm run migrate:supabase:reembed cuando se ingesten documentos jurídicos reales. Costo OpenAI ~$1 por 100k chunks.
  • Migrar archivos de S3 a Supabase Storage. Hoy S3 vacío. Cuando existan, el adapter dual-write (STORAGE_DUAL_WRITE_LEGACY=true) permite migración sin downtime.
  • Habilitar email confirmation en producción. Tras SMTP propio, considerar requerir confirm para nuevos signups (mailer_autoconfirm=false).

Operativa · siguiente sprint

  • Deploy a producción (Vercel). Configurar envs NEXT_PUBLIC_SUPABASE_*, NEXT_PUBLIC_API_URL apuntando al backend productivo, y agregar el dominio de producción a las "Authorized origins/redirect URIs" del OAuth client en Google Cloud.
  • Backend a producción. Migrar de Render legacy al nuevo stack (Render con .env actualizado, Railway, Fly, etc).
  • Decommission de Render Postgres. Tras 14 días sin necesidad de rollback, eliminar instancia y borrar RENDER_DATABASE_URL.
  • Migrar BullMQ workers a pgmq. No urgente; BullMQ funciona con el Redis nuevo. Beneficio: una pieza menos de infraestructura.

Cosmético · sin urgencia

  • Eviction policy volatile-lru en Redis Cloud. No editable en plan Free. Cuando se migre a Essentials/Pro, cambiar a noeviction.
  • 2FA: tabla custom (two_factor_* en public.users) vs auth.mfa_factors de Supabase. Hoy nadie tiene 2FA habilitado, decisión puede esperar.
  • Diseño de iconos PWA. Los actuales son auto-generados (gradiente azul/púrpura + "PL"). Reemplazar por logo definitivo cuando lo tengas.

11Rollback de emergencia (~1 min)

Cuándo: sólo si Supabase tiene caída crítica. Render preservado con paridad de datos (2 users, sin corpus productivo).
  1. Backend .env: comentar AUTH_BACKEND, SEARCH_BACKEND, STORAGE_BACKEND
  2. Backend .env: swap DATABASE_URL ↔ RENDER_DATABASE_URL
  3. Frontend .env.local: comentar NEXT_PUBLIC_AUTH_BACKEND · cambiar NEXT_PUBLIC_API_URL a Render
  4. Frontend: mv src/middleware.ts src/middleware.ts.example
  5. Restart backend + frontend · verificar /login
